ISO14229-1 诊断

Posted on 2017-03-03 In 汽车电子 Views:
本文是对ISO14229-1标准的讲解 Road vehicles — Unified diagnostic services (UDS) — Specification and requirements

汽车中的诊断

什么是诊断，为什么要诊断？
就像人不舒服或得病了去医生，医生会问你哪里怎么不舒服，会做检查来判断是问题所在，这是医生的诊断方法。
汽车也会出现故障，出现故障时如何能快速准确地定位到问题所在，这是汽车中的诊断。
对汽车的诊断又分为在线诊断On Board Diagnostic 与 离线诊断 Off Board Diagnostic。

在线诊断

汽车运行过程中ECU可以实时监控自身和车身运行状况，当出现故障时可以通过故障码的形式实时存储起来，甚至有一定的自我恢复能力。

离线诊断

此时需要特定的诊断设备，专业人员，一般会在4S店，汽车修理厂，确定车身哪里什么ECU出了问题。有时升级或重刷新某些ECU的固件也是诊断的一部分。

几乎所有的前装ECU都需要具备诊断功能，这里的需求是整车厂OEM 提供的。

诊断统一标准

目前对于诊断这一功能，乘用车大部分是ISO14229-1标准，
Road vehicles — Unified diagnostic services (UDS) — Specification and requirements
这样会更方便交流，也提升软件复用率，节省开发成本。
也有一些OEM有自己的诊断标准，我没有接触过，不作讨论。
在网络通信层面上来说 ISO14229-1属于应用层：

如前所述ISO14229 是公路用车统一的诊断服务 Road vehicles — Unified diagnostic services (UDS) 的国际标准。
它分很多部分：

  • Part 1: Specification and requirements
  • Part 2: Session layer services
  • Part 3: Unified diagnostic services on CAN implementation (UDSonCAN)
  • Part 4: Unified diagnostic services on FlexRay implementation (UDSonFR)
  • Part 5: Unified diagnostic services on Internet Protocol implementation (UDSonIP)
  • Part 6: Unified diagnostic services on K-Line implementation (UDSonK-Line)
    The following part is under preparation:
  • Part 7: Unified diagnostic services on Local Interconnect Network implementation (UDSonLIN)
    The titles of future parts will be drafted as follows:
  • Part n: Unified diagnostic services on … implementation (UDSon…

在此文中只介绍 ISO14229-1 UDS Specification and requirements
以下所提到的UDS都指这一标准。

寻址方式

Addressing Types
可分为物理寻址Physical Addressing 和 功能寻址 Functional Addressing
物理寻址时要知道要诊断哪个ECU，知道这个ECU的 ID 号， 如CANID ，然后只这个ECU会响应回复，其它ECU不响应。

功能寻址时发送事先确定好的 ID 号， 所有的ECU都会响应，类似于广播形式。

对于物理寻址和功能寻址时，UDS所规定的负响应方式会有所不同，具体使用时再参考文档。

诊断服务简介

Diagnostic Services
其具体的服务类型大致分为以下几类：

  • 诊断和通信管理功能单元 Diagnostic and Communication Management functional unit
  • 数据传输功能单元 Data Transmission functional unit
  • 传输储存的数据功能单元 Stored Data Transmission functional unit
  • 输入输出控制功能单元 InputOutput Control functional unit
  • 远程激活例程功能单元 Routine functional unit
  • 上传下载功能单元 Upload Download functional unit

以上分别对应 UDS 文档中 9 - 14 章节。

对于具体的服务与细节在此不再详尽介绍，因为当你做过一个项目用到对这些服务就会清楚怎么用了，那时也就很容易理解和记住这些服务。
比如不同的会话模块的切换 DiagnosticSessionControl (0x10)
类似于操作系统中管理员权限和普通用户权限的切换，平时默认的会话状态，当需要更危险的操作比如对硬件的操作如点亮灯，开启雨刮等一般会切换至扩展会话，如果要升级固件又需要切换至编程会话状态，并且还会需要先解锁，是为了安全。

在诊断过程中会读所有故障码DTC或者特定的故障码和历史故障码，ReadDTCInformation (0x19)
就会知道车出现了什么故障和曾给出现过什么故障。当修复好故障码状态会自动清除，有时想更干净会把所有的DTC通过清除所有故障码操作全部清除CIearDiagnosticlnformation (0x14)。
因为故障码会存在于非易失性存储器中不会因断电而丢失 Fault Memory。

诊断所用工具

对于汽车电子软件开发中的诊断工具，做的最稳定强大易用的应当是 Vector 公司的产品，他们产品很全面，
硬件CAN收发器有 VN1610 VN1630 VN1640 等 ：


对应的上位机软件叫 CANoe 其可以模拟整个车身总线的架构，通过其定义好的 DBC 文件可自动解析CAN消息内容 。

可以用 Vector 公司自带的编程语言 CAPL (CAN Application Programming Language ) 来编写自动化的代码来诊断与测试用，极大提升开发效率。
另外对于全面自动化测试诊断（包括ISO14229 , ISO15765 的需求）还可以用 CANoe 额外的一个插件 CANoe.DiVa ，不过是要单独收费。

Vector 的产品还有一个特点就是贵，比如一套带License 可以支持 CANoe 和CAPL编程的 VN1610 或 VN1640 差不多要十来万，这个还是它们产品中相对便宜的。不过我们部门差不多人手一个。

UDS2006与2013版本区别

对ISO14229-1目前常用的有两个版本，2006版与2013版，对于OEM的诊断需求，他们会告诉你要符合ISO14229-1哪一版本的要求，但这两个版本最主要的区别是什么呢？
我把这两个版本文档翻了几遍，其只有一个区别，就是在功能寻址时2006版对 NRC 回应时0x7E和0x7F不抑制suppresse 。
The NRC 0x7F serviceNotSupportedInActiveSession SNSIAS
The NRC 0x7E sub-functionNotSupportedInActiveSessionSFNSIAS

参考
the “IMPORTANT ” in “7.5.1 General definitions”
对 2006 版:
IMPORTANT — As required by the tables in the following subclauses, negative response messages with negative response codes of SNS (serviceNotSupported), SFNS (subFunctionNotSupported) and ROOR (requestOutOfRange) shall never be transmitted when functional addressing was used for the request message.
对 2013 版本:
IMPORTANT — As required by the tables in the following subclauses, negative response messages with negative response codes of SNS (serviceNotSupported), SNSIAS (serviceNotSupportedIn- ActiveSession), SFNS (sub-functionNotSupported), SFNSIAS (sub-functionNotSupportedInActive- Session), and ROOR (requestOutOfRange) shall not be transmitted when functional addressing was used for the request message (exception see Annex A.1 in definition of NRC 0x78).
